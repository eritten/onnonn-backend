================================================================================
CODEX PROMPT: ONNONN — AI-POWERED MEETING PLATFORM BACKEND
Platform Name: Onnonn
Stack: Node.js + Express + MongoDB + Mongoose + Redis + LiveKit + Stripe + OpenAI
================================================================================


PROJECT OVERVIEW
----------------
Build a complete, production-ready backend for Onnonn — an AI-powered video
meeting platform that competes with and surpasses Zoom. Onnonn is built with
Node.js and Express, uses MongoDB as the primary database via Mongoose ODM,
Redis for caching and job queuing, LiveKit for all real-time video and audio
infrastructure, Stripe for billing and subscriptions, and OpenAI for all AI
features. Every feature described in this prompt must be fully implemented.
Nothing should be left as a stub, placeholder, or TODO. The codebase must be
complete, tested, documented, and deployment-ready.


PROJECT STRUCTURE
-----------------
Use a clean feature-based folder structure. Organize code into the following
top-level folders: src/routes, src/controllers, src/services, src/middlewares,
src/models, src/validators, src/utils, src/jobs, src/config, src/ai, and
src/webhooks. The entry point is src/server.js. Use dotenv for environment
variables. Keep all business logic in services. Controllers only handle HTTP
request and response. Never put database queries directly in controllers.


DATABASE
--------
Use MongoDB with Mongoose as the ODM. Define complete, well-structured
Mongoose schemas and models that cover every feature in this prompt. Use
proper references between documents using ObjectId refs where appropriate and
embed sub-documents where embedding is the better MongoDB pattern. Add
indexes on all frequently queried fields. Use Mongoose schema validation,
custom validators, virtual fields, and pre/post hooks where appropriate.

Provide a seed script that populates the database with realistic test data
including users, organizations, meetings, recordings, plans, and
subscriptions.

Define the following Mongoose models at minimum: User, Organization,
OrganizationMember, Department, Meeting, MeetingParticipant, BreakoutRoom,
Poll, PollResponse, QnAQuestion, ChatMessage, Whiteboard, MeetingNote,
Recording, RecordingShare, Webinar, WebinarRegistrant, WebinarAttendant,
Plan, Subscription, Invoice, Contact, ContactRequest, Availability,
AvailabilityBooking, Notification, NotificationPreference, DeviceToken,
AuditLog, AITranscription, AISummary, AIActionItem, AIMeetingCoach,
AISentiment, AIEmbedding, VirtualBackground, CalendarConnection,
MeetingTemplate, Session, PasswordResetToken, EmailVerificationToken,
TwoFactorSecret, SSOConfiguration, GDPRRequest.


================================================================================
AUTHENTICATION AND USER MANAGEMENT
================================================================================

Implement a full authentication system:

User registration with email and password. Hash all passwords with bcrypt.
On registration, create a Stripe customer for the user and save the Stripe
customer ID on the User document. Assign the free plan by default.

Email verification. Send a verification email immediately after registration
with a unique token. Store the token in the EmailVerificationToken collection
with an expiry. The account is inactive until verified. Resend verification
email endpoint.

Login with email and password. Return a signed JWT access token with a short
expiry and a refresh token with a long expiry. Store refresh tokens hashed
in the Session collection and cache active session IDs in Redis.

Refresh token endpoint. Exchange a valid refresh token for a new access token.

Logout endpoint. Delete the session from MongoDB and remove from Redis.
Support logout from all devices simultaneously.

Forgot password. User submits email and receives a reset link with a
short-lived token stored in PasswordResetToken collection. User submits a
new password with the token. Invalidate token after use.

Change password for authenticated users. Require current password
confirmation.

Google OAuth2 login. Allow sign in with Google. If the email matches an
existing user, link and authenticate. Otherwise create a new user. Store
Google OAuth2 tokens on the User document encrypted.

User profile management. Users can update display name, profile photo
uploaded to Cloudinary with the resulting URL stored on the User document,
job title, company name, phone number, timezone, language preference, and bio.

Two-factor authentication using TOTP. Store the TOTP secret in the
TwoFactorSecret collection. Endpoint to generate QR code and secret. Endpoint
to verify and activate 2FA. Endpoint to disable 2FA requiring password
confirmation. Require TOTP code at login when enabled.

Roles: regular user, organization admin, and platform superadmin. Enforce
role-based access control via middleware on every protected route.

Account suspension. Admins and superadmins can suspend or unsuspend accounts.
Suspended users cannot log in. Store suspension status and reason on User.

Session management. List all active sessions from the Session collection for
the authenticated user. Allow revoking individual sessions or all sessions
except the current one.


================================================================================
PLANS AND BILLING
================================================================================

Implement a complete subscription and billing system using Stripe.

Store plan definitions in the Plan collection with fields for name, price,
Stripe price ID, and a limits object containing: max meeting duration in
minutes, max participants per meeting, cloud recording storage in GB, and
flags for each feature.

Free plan limits: meetings up to 40 minutes, maximum 100 participants, no
cloud recording, no AI features.

Pro plan limits: unlimited meeting duration, up to 300 participants, 5GB
cloud recording storage, AI transcription, summary, smart notes, and action
items.

Business plan limits: unlimited meeting duration, up to 1000 participants,
unlimited cloud recording storage, all AI features, SSO, IP allowlisting,
and advanced analytics.

Store the user's current subscription in the Subscription collection with
fields for user reference, plan reference, Stripe subscription ID, Stripe
customer ID, status, current period start, current period end, cancel at
period end flag, and storage used in bytes.

Billing endpoints and logic:

Create a Stripe customer at user registration and store the ID on the User.

List plans endpoint returning all Plan documents with pricing and feature
details.

Subscribe to a plan. Create a Stripe Checkout Session. Return the checkout
URL.

Stripe webhook handler at /webhooks/stripe. Verify webhook signature. Handle:
checkout.session.completed, customer.subscription.created,
customer.subscription.updated, customer.subscription.deleted, invoice.paid,
invoice.payment_failed, invoice.payment_action_required. On each event,
find the user by Stripe customer ID and update the Subscription document.
Store invoice records in the Invoice collection on invoice.paid.

Get current subscription status endpoint.

Cancel subscription at end of billing period.

Upgrade or downgrade plan using Stripe proration.

Billing history. List Invoice documents for the user with amounts, dates,
and Stripe-hosted invoice PDF URLs.

Update payment method via Stripe SetupIntent.

Enforce plan limits everywhere. Check participant limits at meeting creation.
Check storage quota before recording. Check AI feature access before AI
endpoints. Return structured errors with plan limit details.

Usage tracking. Track meeting minutes on the Subscription document. Update
in real time via LiveKit webhooks. Warn users approaching the 40-minute free
plan limit via a LiveKit data message.

MEETINGS
================================================================================

Implement a comprehensive meeting management system.

Meeting model fields: host (User ref), title, description, scheduledStartTime,
expectedDuration, maxParticipants, password (hashed if set), waitingRoomEnabled,
joinBeforeHost, muteOnEntry, allowSelfUnmute, autoRecord, meetingType (enum:
one-on-one, group, webinar), e2eEncryptionEnabled, meetingId (unique short
alphanumeric), joinUrl, liveKitRoomName, status (enum: scheduled, ongoing,
ended, cancelled), recurrence settings object, personal room flag, template
reference, lockedAt, createdAt, endedAt, actualDuration.

Create a meeting. Generate a unique meeting ID. Construct a join URL. Create
a LiveKit room. If auto-record is enabled and the plan allows it, note this
for when the meeting starts.

Generate participant tokens. Verify identity if authenticated. Check password.
Check waiting room. Check plan participant limits. Generate LiveKit token with
role-appropriate grants.

Waiting room. Track waiting participants per room in Redis as a sorted set.
Notify host via LiveKit data message. Endpoints to list waiting participants,
admit one, reject one, admit all.

List meetings for authenticated user with status filter and pagination.

Get meeting by meeting ID.

Update meeting. Notify invited participants of changes by email.

Cancel meeting. Notify participants. Terminate LiveKit room if ongoing.

End meeting. Call LiveKit API to end room. Stop recordings. Mark as ended.
Update actualDuration. Trigger post-meeting analytics job.

Recurring meetings. Store recurrence rule (frequency, interval, days of week,
end date, occurrence count) on the Meeting document. Generate occurrence
documents. Edit single occurrence or all future occurrences.

Personal meeting room. Each User has a personalRoomId field. Separate settings
for personal room stored on the User document.

Meeting templates. MeetingTemplate model stores settings. Endpoint to save,
list, apply, update, and delete templates.

Guest join. Accept display name and optional password. Generate restricted
LiveKit token. Log guest in MeetingParticipant with guestName field.

List current participants via LiveKit API.

Remove participant via LiveKit API.

Mute or unmute participant via LiveKit API.

Transfer host. Update Meeting document host field and re-issue tokens.

Co-hosts. Store co-host user IDs array on Meeting. Endpoints to add and
remove co-hosts.

Lock and unlock meeting. Set lockedAt on Meeting. Block new joins when locked.

Move participant to waiting room mid-meeting via LiveKit API and Redis update.

Raise hand. Track raised hands per room in Redis as an ordered list. Endpoints
to raise, lower, and list.

Emoji reactions. Store each reaction as an embedded document in Meeting.reactions
array with participant ID, emoji, and timestamp.

Breakout rooms. BreakoutRoom model: meeting ref, name, liveKitRoomName,
participantIds array, status. Create rooms, assign participants, generate
breakout tokens, broadcast message to all breakout rooms via LiveKit data
messages, close all and return participants to main room.

Polls. Poll model: meeting ref, question, options array, status, createdBy.
PollResponse model: poll ref, participant ref or guest name, selectedOption.
Endpoints to create, respond, get real-time results, end poll.

Q and A. QnAQuestion model: meeting ref, asker ref or guest name, question,
upvotes array, answeredAt, dismissedAt, answerText, isPublic. Endpoints to
submit, upvote, answer, dismiss, list.

In-meeting chat. ChatMessage model: meeting ref, sender ref or guest name,
content, messageType (public or private), recipient ref if private,
isPinned, deletedAt. Endpoints to send, list public, list private, delete
(host only), pin.

Whiteboard. Whiteboard model: meeting ref, canvasState JSON field, lastUpdatedBy,
updatedAt. Endpoints to get and update canvas state. Broadcast updates via
LiveKit data channel.

Screen sharing control. Track active screen sharer per room in Redis.
Host endpoint to stop screen share via LiveKit API.

Virtual backgrounds. VirtualBackground model: user ref (null for platform
defaults), imageUrl, isDefault. Endpoints to list, upload to Cloudinary,
delete. Enforce per-user limit of ten custom backgrounds.

Spotlight. Track spotlighted participant IDs per room in Redis. Host endpoints
to add, remove, and clear spotlight. Broadcast changes via LiveKit data.

Meeting notes. MeetingNote model: meeting ref, content, createdBy, updatedAt.
Endpoints to add, edit, get, delete for host and co-hosts.

================================================================================
AI FEATURES — POWERED BY OPENAI
================================================================================

All AI features use the OpenAI API. Store the OpenAI API key in environment
variables. Implement all AI logic in src/ai with clean service modules. Each
AI feature must handle errors gracefully and never block the main meeting flow.
All AI results are stored in their respective Mongoose models.

AI MEETING TRANSCRIPTION
After a recording is complete, submit the audio file to OpenAI Whisper API.
Store the full transcription in the AITranscription model with fields for:
meeting ref, recording ref, segments array (each with start time, end time,
speaker label, text, confidence), fullText, language, status. Retrieve
endpoint. Download as plain text or SRT subtitle file endpoint.

AI MEETING SUMMARY
After transcription is complete, use GPT-4o to generate a structured summary.
Store in AISummary model with fields: meeting ref, overview, keyPoints array,
decisions array, actionItems array (each with assignee name, task, deadline),
nextSteps array, status. Retrieve endpoint. Regenerate endpoint.

AI SMART NOTES
Use GPT-4o to extract bullet-point smart notes from the transcription. Store
in AISummary as a smartNotes array. Retrieve endpoint.

AI ACTION ITEM TRACKER
Extract action items from transcription using GPT-4o. Store each in the
AIActionItem model with fields: meeting ref, assigneeName, assigneeUser ref
(if matched to a platform user), task, deadline, completedAt, notes. Endpoints
to list by meeting, mark complete, edit, delete. Send email reminder to
assignees one day before deadline via the reminder job queue.

AI MEETING COACH
After the meeting ends, use GPT-4o to analyze the transcription and generate
a coaching report. Store in AIMeetingCoach model with fields: meeting ref,
speakingTimeBalance object (participant name to percentage), interruptionCount,
offTopicMoments array with timestamps, conciseness feedback, positiveMoments
array, overallScore (0 to 100), summary. Retrieve endpoint.

AI SENTIMENT ANALYSIS
After transcription, use GPT-4o to analyze sentiment per segment. Store in
AISentiment model with fields: meeting ref, overallSentiment (positive,
neutral, negative), overallScore, segments array (each with start time,
end time, sentiment label, score), timeline array for charting. Retrieve
endpoint.

AI CHAT ASSISTANT
In-meeting AI chat endpoint. Accept a question from a participant. Use GPT-4o
with context built from: current meeting transcript buffer stored in Redis,
meeting notes, and any shared documents. Return an AI-generated response.
Also support: summarize so far, list action items so far, draft a response
for the participant. Rate limit per user per meeting.

AI AGENDA GENERATOR
Endpoint accepting meeting topic, duration, participant count, and notes.
Use GPT-4o to generate a structured agenda with timed sections and discussion
points. Store the agenda as a field on the Meeting document. Retrieve and
update endpoints.

AI FOLLOW-UP EMAIL GENERATOR
After meeting ends, use GPT-4o to generate a professional follow-up email
based on summary, action items, and decisions. Store the draft on the Meeting
document. Host can edit and send to all participants via the email queue.

AI TITLE AND DESCRIPTION SUGGESTER
Endpoint accepting keywords or a brief description. GPT-4o returns three
suggested titles and a suggested description for a new meeting.

AI CONTENT SAFETY
Before storing any chat message, call OpenAI Moderation API. If flagged,
reject the message, notify the sender, and log the attempt on the
ChatMessage document with a flagged field. Notify the host. If a user is
flagged more than three times in a meeting, mute their chat for the rest
of the meeting.

AI LANGUAGE TRANSLATION
After transcription, endpoint to translate the full transcription to any
target language using GPT-4o. Store translations on the AITranscription
document as a translations object keyed by language code. Limit to five
translations per meeting per plan.

AI SPEAKER IDENTIFICATION
During transcription processing, use GPT-4o to label speakers in the
transcript using contextual clues. Match speaker labels to participant
names. Store labeled segments on AITranscription. Update speaker labels
endpoint for manual correction.

AI MEETING SEARCH
Semantic search endpoint. Accept a natural language query. Use OpenAI
embeddings to generate a query vector. Store meeting embedding vectors in
the AIEmbedding model with fields: meeting ref, embeddingVector array,
sourceText (summary plus key points). Use MongoDB Atlas Vector Search or
a cosine similarity computation to find the most relevant meetings. Return
ranked results with matched excerpts. Generate and store embeddings after
each meeting summary is created.

AI REAL-TIME CAPTION ASSIST
Endpoint accepting audio chunk as base64 from the frontend. Pass to OpenAI
Whisper. Return caption text. Store caption segments with timestamps in Redis
during the meeting for later transcription assembly.
